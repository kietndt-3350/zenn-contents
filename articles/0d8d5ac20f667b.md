## はじめに ✨

Claude AIを使っていて、「そこまで多く使ったつもりはないのに、なぜか請求額が異常に高い...」と感じたことはないでしょうか。

もし心当たりがある場合、その原因は、セッション内で自分でも気づかないうちに**ほとんど価値を生まない処理にトークンを消費している**ことかもしれません。

Claudeとのやり取りで使われるトークンの多くは、必ずしも新しい内容を生成するためだけに使われているわけではありません。いくつかの統計や実際の観測によると、場合によっては約98.5%ものトークンが、過去のコンテキストを読み直すために使われているとも言われています。

ここでいう過去のコンテキストとは、たとえば以下のようなものです。

- これまでの会話履歴
- 読み込ませたファイル
- instruction
- tool result
- すでに入力済みの情報

つまり、私たちが次のような短い依頼を送ったとしても、

> 「この部分の修正を続けてください」

Claudeはその一文だけを読んでいるわけではありません。「この部分」が何を指しているのかを理解するために、その前にある大量の情報も読み直す必要があります。

そのため、セッションが長くなればなるほど、コンテキストを適切に管理しない限り、コストは想定の何十倍にも膨らむ可能性があります。

この記事では、セッションが長くなるほどClaudeのトークン使用量が増え、作業効率が下がる理由、そしてトークンを節約しながらClaudeと効率よく作業するための5つの習慣について紹介します。

## セッションが長くなるとClaudeの効率が落ちる理由：Context Rot （コンテキストの腐敗）⚠️

**Context Rot** とは、直訳すると「コンテキストの腐敗」という意味です。

セッションが長くなるにつれて、古い要件、すでに修正済みのコード、不要になったログ、途中で却下された方針などがコンテキスト内に残り続けます。  
その結果、Claudeが現在の状況を正しく判断しにくくなり、回答の精度や作業効率が落ちていきます。

たとえば、Claudeとの作業セッションが次のような状態だったとします。

たとえば、Claudeとの作業セッションが次のような状態だったとします。

- 最初に、ドキュメント・仕様書・コードなどを **10,000トークン** 分入力した
- その後、毎回 **100トークン** 程度の短い質問を追加している
- Claudeは毎回 **500トークン** 程度の回答を返している

新しく送っているメッセージだけを見ると、次のように感じるかもしれません。

> 「毎回、短い質問を1つ追加しているだけだから、そこまでコストはかからないはず」

しかし実際には、各ターンでClaudeは、最初に入力された約10,000トークンの情報に加えて、それまでの会話履歴も読み直している可能性があります。

20ターンほどやり取りを続けると、「新しい回答を生成するためのトークン」よりも、「過去の履歴を読み直すためのトークン」の方がはるかに大きくなることがあります。

これが、請求額や usage report を見たときに、多くの人が「思ったよりもかなり使っている」と驚く理由の一つです。

一般的には、コンテキストが長ければ長いほど、AIはより多くの情報を理解でき、情報量が多い分、より効率的に作業できると考えがちです。しかし、実際にはそうとは限りません。

人間の場合、会話や作業の中で「これはもう不要だな」「この方針は採用しないことになったな」と判断した情報は、自然と意識の外に追いやることができます。

一方で、LLMは人間のように不要になったコンテキストをその場で自動的に捨てることが得意ではありません。明示的に `clear` したり、不要な情報を取り除いた新しいセッションに移行したりしない限り、過去の会話や古い情報もコンテキスト内に残り続けます。

そのため、セッションが長くなると、コンテキストには次のような情報が含まれ始めます。

- すでに正しくなくなった古い要件
- すでに修正された古いコード
- 古くなった検証結果
- もはや関係のないログ
- 途中で却下されたアプローチ
- 互いに矛盾する複数のinstruction
- 「うーん」「続けて」「修正して」「そうではないです」といった会話

その結果、Claudeの生産性が下がっていることを示す、次のような兆候が見られるようになります。

- 以前の回答と矛盾する
- ファイルを事前に読まずに、ランダムに修正する
- コンテキストや、すでに合意した内容を忘れる
- 回りくどく、論理的でなく、曖昧な回答をする
- レスポンスが遅くなる

GitHub上の7,000件以上のセッションを分析した研究データによると、セッションが長くなると、Claudeの情報検索精度は92%から7〜8%まで低下し、思考品質は67%低下し、ファイルを事前に読まずに修正する行動は6%から34%に増加したとされています。

まとめると、セッションが長くなるほど、Claudeの作業は雑になり、効率も下がります。そうなると、こちらは何度も聞き直したり、訂正したり、修正を依頼したりする必要が出てきます。そして、そのやり取り自体がさらにトークンを消費します。このループこそが、Context Rotという現象につながる原因です。

## Claudeでトークンを節約し、効率よく作業するための5つの習慣 💡

### 1. Claudeの回答が期待と違う場合は、追加メッセージではなく `/RE` または `/rewind` を使う

Claudeの回答が期待と違っていたときに、

```text
そうではなく、こういうふうにやり直してください
```

のように追加でメッセージを送ると、間違っていた回答もそのままコンテキストに残ってしまいます。

そのため、`/RE` を使って前のメッセージに戻り、誤った部分を削除して、きれいな状態からやり直すようにします。

### 2. コンテキストが60%程度になったら手動でリセットする

Claudeには、コンテキストが95%程度に達したときに自動でcompactする機能があります。

しかし、そのタイミングでは重要な情報の20〜30%程度しか残らない場合があります。

そのため、コンテキストが60%程度に達した時点で、Claudeにこれまで実施した内容を要約してもらいます。

その後、要約をコピーし、`clear` コマンドでコンテキストをリセットします。最後に、新しいチャットにその要約を貼り付けて作業を続けます。

こうすることで、重要な情報を失わずに、不要なコンテキストだけを整理できます。

### 3. Claudeに渡す前に、ファイル形式を目的に応じて整理する

トークン節約の観点では、HTMLやPDFをMarkdown形式に変換してからClaudeに渡すことは有効な場合があります。

HTMLファイルをMarkdown形式に変換すると、読み込み時のトークンを大きく削減できることがあります。PDFファイルの場合でも、不要なレイアウト情報やmetadataを取り除くことで、Claudeが読むべき情報を整理しやすくなります。

ただし、常にMarkdownが最適というわけではありません。

たとえば、Claudeに読ませるだけのレポートや仕様書であれば、Markdownのようなシンプルなテキスト形式の方が扱いやすく、トークン効率もよくなります。

一方で、画面イメージ、インタラクション、図表、デザインシステム、スライド、比較表など、人間が視覚的に確認することも重要な資料では、HTMLの方が有効な場合もあります。HTMLであれば、レイアウトや色、折りたたみ、タブ、簡単な操作などを含められるため、単なるテキストよりも内容を理解しやすくなるケースがあります。

つまり、目安としては次のように考えるとよさそうです。

- AIに読ませることが主目的の資料 → Markdown
- 人間も読む・確認する資料 → HTML
- レイアウトや見た目が重要な資料 → HTML
- 内容だけを抽出したい資料 → Markdown

また、オンライン変換ツールを使う場合は注意が必要です。社内資料、顧客情報、ソースコード、未公開仕様などを外部サービスにアップロードすると、情報漏えいのリスクがあります。

そのため、機密情報を含むファイルは、オンライン変換ツールではなく、ローカル環境で変換するか、利用して問題ないサービスかを事前に確認してから使うのが安全です。

公開情報やサンプルデータであれば、以下のようなオンライン変換ツールを使うこともできます。
ただし、社内資料・顧客情報・ソースコードなどの機密情報には利用しないようにしてください。

- [HTML to Markdown Converter - Code Beautify](https://codebeautify.org/html-to-markdown)
- [HTML/URL To Markdown - DevTool](https://devtool.tech/en/html-md)
- [PDF To Markdown Converter](https://pdf2md.morethan.io/)

### 4. 何度も使う資料はProjectsに入れる

templates、specifications、designなど、何度も使用するファイルはClaudeのProjectsに入れておくと便利です。

Projectsに入れておくことで、Claudeがそれらの情報を再利用しやすくなり、毎回同じ添付ファイルを説明したり、貼り直したりする必要がなくなります。

また、新しいチャットを開始しても、必要なコンテキストを維持しやすくなります。

### 5. 作業をtaskごとに分割する

1つのセッションですべての作業を行うのはおすすめしません。

たとえば、次のような一連の作業を1つの長いセッションで続けるのではなく、

バグ調査 → コード確認 → 修正 → テスト作成 → PR作成 → レビュー → 追加修正

次のように、目的ごとにセッションを分けます。

Session 1: バグの原因調査  
Session 2: 修正の実装  
Session 3: テストの作成  
Session 4: PR descriptionの作成  
Session 5: merge前のリスクレビュー

これは、人間同士で作業するときの進め方にも近いです。

たとえば、コードを書いたあと、そのまま自分の頭の中にある全コンテキストだけでレビューを完結させることはあまりありません。通常は、PR文、差分、関連チケット、確認観点など、レビューに必要な情報を整理したうえで、他のメンバーにレビューを依頼します。

Claudeを使う場合も同じで、1つの長いセッションにすべての作業履歴を残し続けるのではなく、「今やってほしい作業」に必要なコンテキストだけを渡す方が、回答の精度も上がりやすくなります。

各セッションには、その目的に必要なコンテキストだけを含めるようにします。

Context Rotの兆候に気づいたら、現在のセッションを積極的にclearし、新しいセッションで作業を再開することが重要です。

## まとめ

Claudeでは、セッションが長くなるほど過去のコンテキストを読み直す量が増え、トークン使用量も大きくなります。

さらに、不要な情報が増えると Context Rot により回答精度も下がりやすくなります。

大切なのは、**必要な情報だけを整理して渡し、長くなったセッションは早めにリセットすること**です。

これだけでも、トークンの無駄遣いを減らし、Claudeをより効率的に使えるようになります。
