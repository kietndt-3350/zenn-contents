## はじめに ✨

みなさん、こんにちは。

最近では、AIは日常生活だけでなく、私たちの仕事においても**欠かせないツール**になりつつあります。特にソフトウェアエンジニアにとっては、AIを使うことが当たり前になってきました。多くの人がAIを活用しているため、その用途や応用分野も非常に幅広くなっています。

エンジニアである私たちは、AIを使ってコードを書いたり、バグを修正したり、システムアーキテクチャを設計したり、ドキュメントを書いたり、情報を要約したり、さらにはアイデアをブレインストーミングすることさえあります。

しかし、おそらく多くの人が一度は、AIと何時間も「会話」しているのに、返ってくる答えが**どこか的外れだったり、一般的すぎたりして、期待していた結果と違う**と感じた経験があるのではないでしょうか。

> 「AIに聞いたけど、なんか答えが微妙…」

そんなふうに思ったことがある人も少なくないと思います。

実はこのような場合、**AIが賢くないからというわけではないことが多い**です。むしろ、多くの場合は**私たちの質問の仕方（プロンプト）**に原因があります。

まず理解しておきたい大事なポイントがあります。

> **AIは私たちの考えを読み取ることができるツールではありません。**

AIは、**プロンプトに書かれている情報だけを手がかりにして推論し、回答を生成します。**

そのため、プロンプトが曖昧であれば、結果も曖昧になります。

本記事では、私自身がAIと一緒に仕事をする中で調べたり試したりしながら積み重ねてきた経験をもとに、以下の内容について共有したいと思います。

- AIと働くときの考え方  
- 効果的なプロンプトを書くための基本的な考え方  
- AIがより仕事を助けてくれるようなプロンプトの組み立て方  

この文章が、みなさんがAIと**もう少しスムーズに**仕事をするためのヒントになれば嬉しいです。

## 1. プロンプトはミニ仕様書として考える 🧠

AIと一緒に仕事をするときに役立つ考え方のひとつは、  
**「プロンプトを小さな仕様書（specification）のように考える」**ことです。

ソフトウェア開発でも、もし同僚にタスクを依頼するときに次のように言ったとします。

> 「この機能、作っておいて。」

おそらく、依頼された側はすぐにいくつも質問をするはずです。

- この機能は何のためのもの？
- 入力は何？
- 期待する結果は？
- 何か制約や特別な要件はある？

AIに対するプロンプトもこれとよく似ています。  
もし曖昧な指示だけを与えると、AIは**足りない情報を自分で推測するしかありません**。その結果、私たちが期待している答えとは少しずれてしまうことがよくあります。

例えば、次の2つのプロンプトを見てみましょう。

### あまり良くないプロンプト

```
このコードをリファクタリングしてください。
```

AIはこれでも回答できますが、次のような情報が不足しているため、回答はかなり一般的な内容になりがちです。

- このコードがどのレイヤーに属しているのか
- 何を目的にリファクタリングするのか
- どんな制約を守る必要があるのか

### より良いプロンプト

```
Context:
このコードは、RustバックエンドプロジェクトのUseCase層にあるコードです。
アーキテクチャはDDDを採用しています。

Goal:
コードの可読性を改善し、validationロジックを整理したいです。

Constraints:
- 現在の挙動は変更しない
- 関数のインターフェースは変更しない

Output:
- 現在のコードの問題点
- 改善の提案
- リファクタリング後のコード例
```

このように書くことで、AIは次のことを理解できます。

- **Context**：問題の背景や状況
- **Goal**：達成したい目的
- **Constraints**：守るべき条件や制約
- **Output**：期待する出力形式

プロンプトがこのように整理されていると、AIはより**具体的で実用的な回答**を返しやすくなります。

そのため、プロンプトを書くときは上記のようなシンプルな構造を意識すると便利です。

もちろん、毎回この形式を厳密に守る必要はありません。  
ただし、**少しでもコンテキストと目的を明確にするだけで、AIの回答の質は大きく変わることが多い**です。

言い換えると、AIに「質問している」と考えるよりも、  
**「仕様書付きでタスクを依頼している」**と考えるほうが、うまくいくことが多いでしょう。

## 2. 効果的なプロンプトを書くためのいくつかの原則 📖

プロンプトを「小さな仕様書」として考えるとしたら、次に気になるのは  
**どのようにプロンプトを書けばよいのか**という点だと思います。

AIと一緒に仕事をしていく中で、いくつかのシンプルな原則を意識するだけで、AIの回答の質が大きく改善されることに気付きました。  
これらは厳密なルールというよりも、**AIとよりスムーズにコミュニケーションするためのコツ**のようなものです。

ここでは、私が普段よく意識しているいくつかの原則を紹介します。

### 2.1. AIに「役割」を与える

とてもシンプルですが効果的な方法の一つが、**AIに特定の役割を与えること**です。

AIに役割を設定すると、その役割に合わせた視点で回答してくれることが多くなります。

あまり良くないプロンプト:

```
このAPIを設計してください。
```

より良いプロンプト:

```
あなたはRustとDDDに詳しいシニアバックエンドエンジニアです。
次のAPIを設計してください。
```

このように役割を設定することで、AIはより適切な視点から回答しやすくなり、一般的すぎる回答を避けることができます。

### 2.2. 十分なコンテキストを提供する

これはおそらく、**AIと仕事をするときに最も重要な原則**です。

AIは私たちのプロジェクトやコードベース、ビジネスロジックを自動的に理解しているわけではありません。  
必要な情報が提供されなければ、AIは推測するしかなく、結果が期待とずれてしまうことがあります。

例えば、次の2つのプロンプトを見てみましょう。

コンテキストが不足しているプロンプト:

```
このコードを説明してください。
```

より良いプロンプト:

```
これはDDDアーキテクチャを採用したRustバックエンドのrepository層のコードです。
このコードはsqlxを使ってPostgreSQLからデータを取得しています。
このコードの動作を説明してください。
```

また、コンテキストは文章だけでなく、さまざまな方法で提供することができます。

- **関連するファイルやコードを添付する**：ソースコードや関連ファイルを共有することで、AIがシステムの構造を理解しやすくなります。  
- **Markdownの設計ドキュメントを提供する**：仕様書やユーザーストーリーなどのドキュメントは、ビジネスロジックの理解に役立ちます。  
- **構造化データ（JSONやスキーマなど）を提供する**：データの構造を正確に理解させることができます。  
- **画像や図を利用する**：UIのスクリーンショットやアーキテクチャ図などは、文書よりも早くコンテキストを伝えられる場合があります。

まとめると、次のように考えると分かりやすいでしょう。

> **AIは、与えられるコンテキストが多いほど、より正確な回答を返しやすくなります。**

### 2.3. 期待する出力形式を明確にする

もう一つのシンプルで効果的な方法は、**どのような形式で回答してほしいかを明確にすること**です。

例えば次のように指定します。

```
次の形式で回答してください。

1. 問題点
2. 解決案
3. コード例
```

このように出力形式を指定すると、回答が整理され、読みやすく実用的になります。

---

### 2.4. 問題を小さく分割する

AIは、大きく複雑なタスクよりも、**小さく分割されたタスク**を処理する方が得意です。

あまり良くない例:

```
このシステムのアーキテクチャをすべて設計してください。
```

より良い例:

```
Step1: domain modelを設計する
Step2: repository interfaceを設計する
Step3: usecaseを設計する
```

問題を分割することで、AIは各ステップに集中でき、結果もよりコントロールしやすくなります。

### 2.5. AIの回答は必ずレビューする

AIは便利なツールですが、常に正しいとは限りません。

- コンテキストを誤解する  
- 不正確な情報を含む  
- 自信を持って間違える  

といったこともあります。

そのため、AIの回答は必ず確認し、必要に応じてコードや環境で検証することが重要です。

AIは優秀なコラボレーターですが、**最終的な判断はエンジニアが行うべきものです。**

## まとめ ✨

本記事では、AIとより効果的にコミュニケーションするための考え方として、**プロンプトを「ミニ仕様書」として捉える視点**と、実際にプロンプトを書く際に意識しているいくつかの原則を紹介しました。 

AIは非常に強力なツールですが、**魔法の箱ではありません**。  
AIの能力を最大限に活かすためには、私たち自身が「どのように質問し、どのように指示するか」を工夫する必要があります。

言い換えると、**Prompt Engineering もまたエンジニアリングスキルの一つ**だと言えるでしょう。

私自身もまだ試行錯誤しながらAIと仕事をしていますが、本記事がみなさんがAIとよりスムーズに協働するためのヒントになれば嬉しいです。

もし「こんなプロンプトの書き方が効果的だった」「こういうAIの使い方をしている」といった経験があれば、ぜひ共有していただけると嬉しいです。
